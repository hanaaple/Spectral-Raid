AbilitySystem — GAS-like

UE5 Gameplay Ability System(GAS) 개념을 Unity에 맞게 경량화한 수치·이펙트 관리 시스템. Assets/Scripts/Core/AbilitySystem/

🎯 설계 의도 — 장비·스킬·디버프 등 수십 개의 출처가 수치를 동시에 변경할 때, 각 시스템이 필드를 직접 조작하면 적용 순서 의존성과 잔류 값 문제가 발생한다. GAS-like는 모든 수치 변경을 GameplayEffect 단일 채널로 추상화해 적용·해제가 서로 독립적으로 동작하도록 보장한다.
Attribute 시스템

레이어 구조

// ── 에셋 — 초기값 정의 (Inspector에서 편집) ──────────────────────
AttributeInitData (ScriptableObject)
└── AttributeSetInitData[] — 어떤 AttributeSet을 어떤 값으로 초기화할지
    └── AttributeFieldInitData[] — { fieldName, BaseValue } 쌍
// ── 런타임 — 수치 저장소 ─────────────────────────────────────────
AttributeSet (abstract class) — 어트리뷰트 컨테이너. 확장 시 서브클래스 추가
├── CharacterAttributeSet { Health, MaxHealth, Stamina, MaxStamina, Speed }
└── CombatAttributeSet { Damage }
// ── 식별자 / 데이터 단위 ──────────────────────────────────────────
AttributeHandle (readonly struct) — SetType + fieldName + FieldInfo 캐싱
AttributeData (struct) — BaseValue / CurrentValue 쌍
// ── 진입점 ───────────────────────────────────────────────────────
AbilitySystemComponent (MonoBehaviour) — Dictionary<Type, AttributeSet> 보유

AttributeData — BaseValue / CurrentValue 이중 구조

필드의미변경 주체
BaseValue 수치의 원본값. 영구 변경에 해당. Effect 해제 후에도 유지된다. Instant GE, SetBaseAttributeValue(), Periodic GE
CurrentValue 활성 GE 모디파이어를 BaseValue에 누산한 결과값. UI·전투 판정에 사용. ASC — Effect 적용·해제 시 자동 재계산 (dirty-on-write)
📌 dirty-on-write 전략 — CurrentValue는 Effect 적용·해제 시점에 한 번만 재계산한다. 읽기(UI 갱신·전투 판정)는 캐싱된 값을 그대로 반환하므로 호출 빈도가 높아도 비용이 없다. 반대로 recalculate-on-read 방식은 읽기 호출마다 활성 Effect 전체를 순회해야 한다.

AttributeHandle — Reflection 비용 격리

GE Modifier는 "어떤 어트리뷰트를 대상으로 할지"를 AttributeHandle로 지정한다. 핸들은 생성 시 1회 Reflection으로 FieldInfo를 캐싱하고, 이후 런타임 읽기·쓰기는 캐싱된 참조만 사용한다.

C# — AttributeHandle 선언 패턴 
// static readonly 선언 → 앱 수명 동안 FieldInfo를 단 한 번만 resolve
public static readonly AttributeHandle Health =
    new(typeof(CharacterAttributeSet), nameof(CharacterAttributeSet.Health));

// 읽기·쓰기는 캐싱된 FieldInfo를 통해 수행 — 런타임 string 탐색 없음
float hp = asc.GetAttributeCurrentValue(Health);

SO 초기화 흐름 — 코드 없이 초기 스탯 설정

캐릭터마다 다른 초기 스탯을 코드 변경 없이 Inspector에서 관리한다. Awake()에서 Reflection으로 인스턴스를 생성하고 필드를 세팅한다.

AttributeInitData (SO — Inspector에서 할당)
└── attributeSets[ ]
    └── typeName: "CharacterAttributeSet, Assembly..."
        └── attributes: [ { "Health", 100 }, { "Speed", 5 }, ... ]
▼ AbilitySystemComponent.Awake()
Type.GetType(typeName) → Activator.CreateInstance()
→ FieldInfo.SetValue(set, AttributeData(baseValue)) // 각 fieldName별 세팅
→ AddAttributeSet(set) // _spawnedAttributeSets[Type] = set
GameplayEffect 시스템

에셋 / 런타임 레이어 분리

GE는 불변 정의(ScriptableObject 에셋)와 적용 인스턴스(런타임 Spec)로 철저히 분리된다. SO는 여러 캐릭터가 공유해도 상태가 섞이지 않고, Spec은 적용 시점의 Level·Context를 스냅샷해 독립적으로 관리된다.

// ── 에셋 — 불변 정의 (Inspector에서 편집) ──────────────────────
GameplayEffect (ScriptableObject)
├── type: GameplayEffectType
├── duration, period, executePeriodicEffectOnApplication
├── modifiers: GameplayModifier[] ← Serializable struct
└── executions: GameplayEffectExecution[] ← SO 참조 (추상 클래스)
GameplayModifier (Serializable struct — GE에 포함)
├── attributeSetTypeName + fieldName → 런타임에 AttributeHandle로 변환
├── operation: GameplayModifierOperation
└── magnitude (현재 ScalableFloat 고정값 — TODO: Level 커브 테이블)
// ── 런타임 — 적용 인스턴스 ──────────────────────────────────────
GameplayEffectSpec (class — ApplyEffect 시 생성)
├── Definition: GameplayEffect ← 원본 SO 참조 (불변)
├── Context: GameplayEffectContextHandle ← Instigator ASC, EffectCauser
├── Level: float
└── Modifiers: GameplayModifierSpec[] ← 생성 시 Handle resolve + Magnitude 계산 완료
GameplayModifierSpec (readonly struct — Spec에 포함)
├── Handle: AttributeHandle ← resolve 완료 (FieldInfo 캐싱)
├── Operation: GameplayModifierOperation
└── EvaluatedMagnitude: float ← Level 기반 계산 완료
ActiveGameplayEffect (class — _activeEffects Dictionary에 등록)
├── Handle: ActiveGameplayEffectHandle ← 제거 키 (int ID 래퍼)
├── Spec: GameplayEffectSpec
├── RemainingDuration ← Duration 타입 전용
└── PeriodTimer ← 마지막 주기 실행 이후 경과 시간
📌 Spec 생성 시 Modifier를 미리 resolve하는 이유 — GameplayModifier는 대상 어트리뷰트를 문자열(AssemblyQualifiedName + fieldName)로 직렬화한다. 이를 AttributeHandle(FieldInfo 캐싱)로 변환하는 비용이 있으므로, Spec 생성 시점에 단 1회만 수행하고 이후 Tick·재계산에서는 캐싱된 핸들만 사용한다. GE SO 하나가 수십 개의 Active Effect로 동시에 적용되더라도 resolve 비용은 각 Spec마다 1회뿐이다.

GameplayEffectType

Instant
BaseValue를 즉시 영구 변경 후 종료. Active Effect로 등록되지 않으므로 Handle 없음.
예: 체력 회복, 데미지
Infinite
명시적 Remove 전까지 CurrentValue에 지속 반영.
예: 장비 스탯 버프, 패시브
Duration
지정 시간 후 자동 해제. Period 설정 시 주기마다 추가 실행.
예: 버프·디버프, DoT·HoT

Apply 흐름

흐름도 
ApplyGameplayEffectToSelf(GE so, context, level)
  ▶ new GameplayEffectSpec(so, context, level)
      // GameplayModifier → GameplayModifierSpec 변환 (Handle resolve + Magnitude 계산)

  // ── Instant ───────────────────────────────────────────────────
  ▶ ExecuteModifiers(spec)   // BaseValue 직접 변경 → CurrentValue 재계산
  ▶ return ActiveGameplayEffectHandle.Invalid

  // ── Duration / Infinite ──────────────────────────────────────
  ▶ new ActiveGameplayEffect(handle, spec)
  ▶ _activeEffects.Add(handle, active)
  ▶ Period > 0 && ExecuteOnApplication  → ExecuteModifiers()       // 즉시 1회 실행
  ▶ Period == 0                          → RecalculateAffectedAttributes()  // CurrentValue 반영
  ▶ return handle

// ── Tick (매 프레임) ─────────────────────────────────────────────
TickActiveEffects(deltaTime)
  ▶ Period > 0  → PeriodTimer 누산 → 주기 도달 시 ExecuteModifiers()
  ▶ Duration    → RemainingDuration 감산 → 0 이하 시 RemoveActiveGameplayEffect(handle)

Period — 주기 실행 (DoT / HoT)

⏱ Period > 0이면 Tick마다 경과 시간을 누적하고 주기 도달 시 ExecuteModifiers()를 호출한다. 이때 Modifier는 CurrentValue가 아닌 BaseValue를 직접 변경하므로 독(DoT)·회복(HoT)처럼 원본 수치를 반복 수정하는 용도에 적합하다. ExecutePeriodicEffectOnApplication = true면 적용 즉시 1회 선행 실행된다.

ModifierOperation — CurrentValue 계산 공식

수식 
CurrentValue = ((Base + ΣAddBase) × MultiplyAdditive / DivideAdditive × ΠMultiplyCompound) + ΣAddFinal

// Override가 하나라도 있으면 위 계산 전체를 무시하고 해당 값으로 덮어씀
Operation누산 방식용도 예시
AddBaseaddBase += mag플랫 수치 증가 (+50 공격력), Instant / Periodic에서 주로 사용
MultiplyAdditivemultiplyAdditive += (mag − 1)퍼센트 증폭 합산 (10% + 20% = 30% — 소스가 많아도 단순 합산)
DivideAdditivedivideAdditive += (mag − 1)퍼센트 감쇄 합산 (방어력 감소 중첩)
MultiplyCompoundmultiplyCompound *= mag복리 배율 (1.1 × 1.2 = 1.32 — 아이템 세트 보너스 등)
AddFinaladdFinal += mag배율 계산 이후 플랫 합산 (최종 고정 보정값)
Override최종 고정값 강제 부여스턴 시 Speed = 0, 특정 상태 고정

GameplayEffectContext — 발동 맥락

GE 적용 시 발동 주체(Instigator ASC)와 실제 원인 오브젝트(EffectCauser)를 함께 전달할 수 있다. GameplayEffectContextHandle은 null-safe struct 래퍼로, 맥락이 불필요한 경우(장비 장착 등) default로 전달한다. Execution에서 Source/Target ASC를 모두 읽을 수 있어 발사체 피해 계산처럼 발동 주체의 수치를 참조해야 하는 경우에 활용된다.

C# — Context 사용 예시 
// 발사체가 대상 ASC에 피해 GE를 적용하는 경우
var ctx    = new GameplayEffectContext(instigatorAsc, bulletGameObject);
var handle = new GameplayEffectContextHandle(ctx);
targetAsc.ApplyGameplayEffectToSelf(damageGE, handle);

// 장비 장착처럼 맥락이 없는 경우 — default(struct) 전달
asc.ApplyGameplayEffectToSelf(equipmentStatGE);

GameplayEffectExecution — 예정

🔧 미구현 (구조 확정, 구현 예정) — 단순 Modifier로 표현할 수 없는 복합 계산 (예: Source.Damage − Target.Armor × 관통률)을 GameplayEffectExecution ScriptableObject 서브클래스로 캡슐화한다. Execute(parameters) 안에서 Source·Target ASC를 읽고 parameters.AddOutput(handle, magnitude)으로 결과를 출력하면 ASC가 AddBase / Override 연산으로 BaseValue에 즉시 반영한다. GE 하나에 여러 Execution을 붙일 수 있으며 각 출력은 순서대로 처리된다.

  • GameplayAbility (GA) 구현 — 스킬·액션을 GE와 연결하는 실행 단위
    • · ActivateAbility() / EndAbility() 진입·종료 생명주기
    • · 코스트(Stamina 소모) / 쿨다운(Duration GE)을 GE로 표현
    • · AbilitySystemComponent에 GA 목록 등록·슬롯 관리
    • · 입력 바인딩: Skill_Q / Skill_R 액션과 GA 슬롯 연결
  • GE 스택 구현 — 같은 GE를 중복 적용할 때 스택 수 관리 및 Overflow 정책 결정
  • Gameplay Cue 구현 — GE 적용·해제 시 VFX·SFX 트리거 연결

  • ItemData 모듈 조합 구조 정의
    • · uid, displayName, icon은 공통 필드로 유지

    • · Particle System처럼 필수 모듈 + 선택 모듈 구조로 구성
  • 1차 ItemModule 타입 구현
    • · EquipmentModule: 장착 슬롯, 장착 프리팹
    • · WeaponGripModule: 한손 전용, 양손 전용, 한손/양손 전환 가능
    • · SlotBlockModule: 장착 시 차단할 슬롯
    • · StatModifierModule: AttributeSystem에 적용할 스탯 보정
  • ItemArchetype 기반 필수 모듈 검증
    • · Weapon: EquipmentModule, WeaponGripModule 필수
    • · Armor, Accessory, MeleeSubWeapon: EquipmentModule 필수
    • · Archetype과 EquipmentSlot 불일치 시 경고 또는 장착 실패
  • 장비 슬롯 규칙 구현
    • · Weapon 1개
    • · Armor 1개
    • · Accessory 4개
    • · MeleeSubWeapon 1개, 근접 보조 무기 한정
  • EquipmentComponent.Equip(ItemData) 구현
    • · EquipmentModule이 없는 아이템은 장착 실패
    • · 슬롯 최대 수량 검사
    • · SlotBlockModule 기반 장착 충돌 검사
    • · 단일 슬롯은 기존 장비 자동 교체, 장신구 5번째 장착은 실패
    • · 장착 프리팹 생성 및 손/장착 위치 Attach
  • 장착 상태와 활성 상태 분리
    • · 한손/양손 전환 가능 무기는 양손 상태에서 MeleeSubWeapon 비활성
    • · 비활성 장비는 장착 상태를 유지하되 전투 사용 또는 스탯 적용에서 제외 가능
